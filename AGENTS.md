# AGENTS.md

Guidance for AI coding agents working on the **Emotion & Culture Lab** website
(`ZJU-ECLab.github.io`). Read this before making changes. For a human-oriented
overview, see `README.md`; this file goes deeper into how the system fits
together and the conventions you must follow.

---

## 1. What this project is

A **static site** for the Emotion & Culture Lab at Zhejiang University. It has
two parts that share a visual language but are otherwise independent:

1. **The lab site** — Home, News, People, Alumni, Publications, Resources,
   Courses, Join Us, Contact. Generated by `build.py` from Markdown + YAML.
2. **The weekly journal SPA** (*《东西情报》*) — a self-contained client-side
   app at `/journal/` that reads JSON produced by an external pipeline.

There is **no framework, no bundler, no npm**. The build is a single Python
script that renders Jinja2 templates. Output goes to `dist/` (git-ignored) and
is published to GitHub Pages.

---

## 2. Build & verify (do this after every change)

```bash
# With Nix (matches CI exactly — preferred):
nix run                                   # build into dist/
nix develop --command python3 build.py    # equivalent

# Without Nix:
pip install jinja2 markdown pyyaml
python3 build.py
python3 build.py --serve                  # build, then serve dist/ on :8000
```

**Always rebuild after editing** content, templates, `build.py`, or assets. The
build **fails loudly if any `/assets/...` reference has no matching file**
(`validate_assets()`), so broken image/PDF paths are caught before they ship.
Treat a non-empty "broken asset reference(s)" warning as a failure to fix.

There is no separate test suite. Verification = a clean build with no broken
references, plus a visual check via `--serve` when you change layout or styles.

## 4. How the build works (`build.py`)

`build()` runs these steps in order:

1. `clean_dist()` — wipe and recreate `dist/`.
2. `copy_assets()` — copy `assets/` → `dist/assets/` verbatim.
3. `copy_static()` — write `.nojekyll`; copy `CNAME` if present.
4. `build_pages(env)` — render every `content/pages/**/*.md` (includes `home.md`).
5. `build_data_page(...)` ×4 — render People, Alumni, Publications, Courses.
6. `build_news(env)` — render the news list + each post.
7. `build_404(env)` — render `404.html`.
8. `copy_journal()` — copy the entire `journal/` directory (SPA shell + data) to `dist/journal/`.
9. `validate_assets()` — fail loudly on broken `/assets/...` references.

### Content model

- **Markdown pages/news**: `parse_markdown()` splits YAML front-matter (between
  `---` fences) from the Markdown body, then renders the body with the `markdown`
  library (extensions: `extra`, `sane_lists`, `smarty`, `attr_list`).
- **YAML data**: `load_yaml()` for list pages. Publications get an extra
  `_group_publications()` transform that groups items by `year` (newest first)
  and adds a `total` count.

### Clean URLs

`write_page("/about/")` writes `dist/about/index.html`. `page_url()` maps
`content/pages/contact.md` → `/contact/`, and `home.md` → `/`.

### Custom Jinja filters (registered in `make_env()`)

| Filter | Purpose |
|--------|---------|
| `inline_md` | Safe inline-markdown subset: `**bold**`, `*italic*`, `[text](url)`. HTML-escapes input first, so data can't inject markup. Use for author/venue strings. |
| `initials` | Two-letter initials for the photo-fallback avatar. |
| `safe_html` | Pass-through for trusted Markdown-rendered HTML (identity filter). |

### Per-page accent theming (important)

The whole UI re-tones from a single `--accent` color. `build.py` defines:

- `ACCENTS` — a per-section ROYGBIV palette (`home`=red … `contact`=purple).
- `COMPLEMENTS` — the roughly opposite hue, for dual-tone surfaces.

These are injected into `base.html` as inline CSS variables:
`:root { --accent: ...; --complement: ...; }`. Page front-matter may override
with an `accent:` key. **If you add a new top-level page, add its slug to both
`ACCENTS` and `COMPLEMENTS`** or it falls back to `DEFAULT_ACCENT`.

## 5. Templates

All templates extend `base.html`, which provides the `<head>`, app bar + nav,
footer, image lightbox, and back-to-top FAB, and exposes `{% block content %}`.

| Template | Rendered for |
|----------|--------------|
| `base.html` | Shared shell (don't render directly) |
| `home.html` | `/` — hero "emotion moodboard" + quick-nav grid |
| `page.html` | Generic prose pages (default template) |
| `news-list.html` / `news-post.html` | `/news/` and individual posts |
| `people.html` / `alumni.html` | `/people/`, `/alumni/` |
| `publications.html` | `/publications/` — year-filter chips + grouped list |
| `courses.html` | `/courses/` |
| `resources-index.html` | `/resources/` |
| `404.html` | Not-found page |

Navigation is **not** in the templates — it's the `NAV` list in `build.py`
(dropdowns via `children`). `QUICK_NAV` drives the homepage card grid. Edit those
Python lists to change site navigation.

A page selects its template via front-matter `template:`; otherwise `page.html`.

## 6. CSS architecture (`assets/tokens.css` + `assets/site.css`)

Material Design 3 "Expressive" system. **Two-tier tokens**:

1. **Shared semantic tokens** live in `assets/tokens.css`, loaded *before* both
   `site.css` (lab site) and `style.css` (journal) in their respective HTML
   shells. These are the 34 M3 roles that derive from `--accent` via
   `color-mix()` and are identical across both apps (surfaces, ink, outlines,
   containers, shape scale, elevation, motion curves).
2. **Source + app-specific tokens** in each file's `:root`: `site.css` keeps the
   `--logo-*`, `--emotion-*`, `--paper-*`, `--tertiary*`, `--radius-*`,
   `--shape-expressive-*`, gradient tokens, and its own `--accent`, `--brand`,
   `--headline-size`, `--w`. The journal keeps its own `--accent` and layout
   width. Dark mode overrides live in `[data-theme="dark"]`.
3. **Component classes** consume only semantic tokens.

> **Why a separate `tokens.css`?** The lab site and journal are separate apps
> (see §7) but share an identical M3 foundation. Extracting only the *provably
> identical* tokens removes duplication without coupling the two apps'
> distinctive parts. **Tokens that intentionally differ — `--accent`, `--w`,
> `--headline-size`, `--brand` — stay in each app's own file.** Don't move them
> into `tokens.css`.

### Token conventions — read before "cleaning up"

The token block defines several **complete, intentional design palettes** that
are a deliberate API, even when not every entry is currently referenced:

- `--logo-*` (brand colors), `--emotion-*` (emotion spectrum)
- `--shape-*`, `--radius-*`, `--shape-expressive-*` (shape scales)
- `--elev-*`, `--ease-*`, `--state-*`, M3 roles like `--tertiary*`

**Do not delete individual tokens from these scales just because grep shows zero
uses.** Removing one entry from a named scale makes the system inconsistent and
hurts maintainability. Only remove a token if you're removing the whole feature
it belongs to.

### Gradient note

`--gradient-primary` is a monochromatic accent gradient
(`color-mix(in oklch, var(--accent), black/white …)`). It is intentionally
derived from the accent so it stays harmonious on every page. Earlier it blended
the accent with a hardcoded pink, which went muddy on non-warm accents (e.g. the
green Publications page) — don't reintroduce a fixed second color.

### Dynamic CSS variables (set from JS)

`site.js` / `app.js` set per-element vars at runtime via `style.setProperty`:
`--ripple-x/y`, `--tilt-x/y`, `--em-*` (floating emoticons), `--enter-i`,
`--plate-shape`, `--accent` (journal). Don't assume a `--foo` is dead just
because it's absent from CSS source — check the JS too.

### Style conventions

- 2-space indentation; lowercase hex; `color-mix()` and `oklch()` are used
  freely (target browsers are modern, ~2025+).
- Component classes use M3-ish BEM: base `.pub-card`, modifier `.chip--selected`.
- Hover effects for pointer devices live in the `@media (hover: hover)` block
  near the end of the file — put new hover rules there, not inline.

## 7. JavaScript

Plain ES5-style IIFEs, no modules, no build step. 2-space indent, single quotes,
`'use strict'`.

- **`site.js`** — lab-site interactions: theme toggle (persisted in
  `localStorage`), mobile drawer, scroll progress bar, scroll-reveal
  (IntersectionObserver), button ripple, 3D tilt cards, hero moodboard, image
  lightbox, alumni carousel, floating emoticons, back-to-top FAB, dropdown menus.
- **`shape-morph.js`** — applies organic clip-path polygon shapes to hero/nav
  icons and morphs them on hover.
- **`app.js`** — the journal SPA only: hash routing (with View Transitions when
  supported), landing page, issue view, search/filter, scroll-spy. It defines
  its own theme toggle, progress bar, scroll-reveal, and FAB (parallel to
  `site.js` — they're separate apps; see below).

> **`route` monkey-patching in `app.js`**: `route` is intentionally re-wrapped
> twice (reveal-on-navigation, then FAB-reset). Preserve this layering if you
> touch it — each wrapper calls the previous `route`.

### Shared-code duplication is by design

`site.js`/`site.css` (lab site) and `app.js`/`style.css` (journal) duplicate
some tokens and helpers. This is **intentional**: the journal is a standalone
SPA whose data is produced and deployed by an external pipeline, so it must not
depend on lab-site files. Don't "DRY them up" into shared modules without
understanding that coupling.

## 8. Editing content (most common tasks)

- **Add/edit a publication**: edit `content/data/publications.yml`. Wrap lab
  authors in `**bold**`; italicise venue parts with `*asterisks*`; `doi` is a
  full URL; `pdfs` is a list of `{ href, label }` (label defaults to "PDF").
  Items auto-group by `year`. Put the PDF in
  `assets/files/publications/` and reference it with an absolute `/assets/...`
  path.
- **Add a news post**: create `content/news/YYYY-MM-DD-slug.md` with front-matter
  (`title`, `date`, `category`, `slug`, optional `thumbnail`, `images`,
  `excerpt`) + Markdown body. Posts sort newest-first by date.
- **Add a person**: edit `content/data/people.yml` (grouped by role). Photos go
  in `assets/img/people/`; a missing photo falls back to initials.
- **Add an image/document**: drop it under `assets/img/` or `assets/files/` and
  reference it with an absolute path. The build verifies it exists.

## 9. Conventions summary

| Area | Convention |
|------|------------|
| Python (`build.py`) | 4-space indent, double quotes, snake_case, `UPPER_SNAKE` constants, PEP 8 |
| CSS | 2-space indent, lowercase hex, semantic tokens, hover rules in the `@media (hover)` block |
| JS | 2-space indent, single quotes, `'use strict'`, IIFEs, no modules |
| HTML/Jinja | 2-space indent, double-quoted attributes |
| YAML | 2-space indent |
| URLs | Clean URLs (`/people/`), absolute `/assets/...` paths |
| Decorative images | `alt=""` + `aria-hidden` on the wrapper; meaningful images get real `alt` |

## 10. Guardrails for agents

- **Rebuild and check for broken-asset warnings** after every change. A build
  warning about missing `/assets/...` references means something is wrong —
  fix it before finishing.
- **The journal SPA (`app.js`, `style.css`, `journal/`, `data/`) is owned by an
  external pipeline.** Avoid changing its data contract. Visual tweaks are fine;
  structural/data changes can break the ECLab-News integration.
- **Never edit `dist/`** — it's generated. Edit sources and rebuild.
- **Don't add build tooling (npm, bundlers, frameworks)** without an explicit
  request. The zero-dependency, single-script design is a deliberate choice.
- **Adding a top-level page?** Add its slug to `ACCENTS` and `COMPLEMENTS` in
  `build.py`, and add it to `NAV` (and `QUICK_NAV` if it belongs on the home
  grid).
- **Be conservative with design tokens** — they're a documented system, not
  dead code (see §6).
- **Accent colors differ between lab site and journal on purpose**
  (`DEFAULT_ACCENT` in `build.py` vs `app.js`) — they're separate apps with
  different brand hues. Don't "unify" them.
